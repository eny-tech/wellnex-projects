# Decision: WellNex Azure IaC lives in a new `wellnex-infra` repo

| Field | Value |
|-------|-------|
| Status | **Accepted** — 2026-04-19 |
| Decided in | Session `cloudgov`, continuation of Phase 1 of the [Cloud Governance](../README.md) project |
| Supersedes | Implicit "IaC lives in `wellnex-server/infra/azure/`" arrangement from 2026-04-18 |
| Owner | Cloud Governance project |
| Related | [2026-04-19-azure-deploy-bootstrap.md](../plans/2026-04-19-azure-deploy-bootstrap.md), [2026-04-19-cloud-naming-migration.md](../plans/2026-04-19-cloud-naming-migration.md) |

## Context

The first cut of WellNex Azure IaC landed inside `wellnex-server/infra/azure/`
on 2026-04-18 because the API was the first deployable component. The
configuration immediately grew past that scope: it provisions the Static Web
App for the frontend (`wellnex` repo), the Container Apps environment shared
across services, ACR and Key Vault used by all systems, PostgreSQL, and the
managed identity wiring. None of that belongs to one application repo.

Concurrently, the [Cloud Resource Naming standard](https://github.com/eny-tech/wellnex-docs/blob/main/governance/cloud-resource-naming.md)
was promoted to v1.2.0 and reserves `system=iac` as a peer of `web`, `api`,
`agents`, `obs`, `data`, `sec`, `core`. The standard treats infrastructure
as a first-class system, not a sub-folder of an application.

The operator also asked for an "online state file" — meaning the Terraform
state must move to an Azure Storage Account. That requires a one-shot
**bootstrap** that itself is IaC, and it has no natural home inside an
application repo.

## Decision

Create a new GitHub repository **`eny-tech/wellnex-infra`** and migrate
the entire Azure Terraform configuration into it. Application repositories
(`wellnex`, `wellnex-server`, `wellnex-health-agents`) will not contain
IaC after the cutover; they contribute container images, build artifacts,
and configuration values consumed *by* the IaC.

## Rationale

1. **Naming standard already says so.** `system=iac` is the assigned home for
   "state, terraform backends, automation". A standard that names IaC as a
   distinct system but leaves it physically inside another system's repo is
   self-contradictory.
2. **Scope mismatch is already real.** The current `wellnex-server/infra/azure/`
   provisions resources for `web`, `api`, `agents`, `data`, `sec`, `obs`,
   and `core`. Calling that "the server's infra" is a misnomer that will
   keep biting reviewers and on-call.
3. **Bootstrap needs its own root.** State storage, OIDC federation, and
   resource-provider registrations are once-per-subscription, not once-per-app.
4. **Growth path.** Per-environment compositions (`envs/tst/`, `envs/stg/`,
   `envs/prd/`) and reusable modules (`modules/api-stack/`, `modules/web-stack/`,
   `modules/naming/`) are textbook Terraform layouts. Forcing them under
   `wellnex-server/` would distort the server repo and pull infra reviewers
   into unrelated PRs.
5. **Cadence mismatch.** Application code ships continuously. Infrastructure
   ships cautiously, behind manual approval, with blast-radius gating. They
   want different CI workflows, different reviewers, different protection
   rules.
6. **Mirrors the just-validated split.** `wellnex-projects` was spawned as a
   peer to `wellnex-docs` for exactly the same separation-of-concerns reason.
   IaC deserves the same treatment.

## Anti-goals

- **No** per-system IaC repos (`wellnex-infra-api`, `wellnex-infra-data`,
  …). One platform, one infra repo. Per-system slicing is achieved with
  **modules** inside `wellnex-infra`, not with separate repos.
- **No** lingering IaC in application repos after cutover. `wellnex-server/infra/`
  is deleted or reduced to a pointer README; same for any future
  application-repo `infra/` directories.
- **No** infra logic in `wellnex-docs` or `wellnex-projects`. Those repos
  hold the *standards* and *project state*, not the executable Terraform.
- **No** secrets in `wellnex-infra`. Secrets go to Key Vault at apply time;
  GitHub repo secrets hold only OIDC client/tenant/subscription IDs.
- **No** dual-maintenance window. The cutover is a single PR pair: a PR that
  imports state into `wellnex-infra` and a PR that deletes the old config
  from `wellnex-server`. We do not run both for any length of time.

## What this decision does *not* cover

- The naming-function (a separate Function App) is being open-sourced into
  its own repository; see [cloud-naming migration](../plans/2026-04-19-cloud-naming-migration.md).
  `wellnex-infra` will *consume* that service, not host it.
- The exact module boundaries inside `wellnex-infra/modules/` are open to
  refinement during the carve-out; see the bootstrap plan for the starting
  cut.

## Implementation triggers

The decision is final, but most execution waits on:

| Trigger | What unlocks |
|---------|--------------|
| Operator creates `eny-tech/wellnex-infra` on GitHub | Repo scaffold + first push |
| Operator confirms Azure subscription is created | `bootstrap/` apply (state account, OIDC) |
| Operator authenticates `az login` (or supplies SP creds) inside the devcontainer | Anything `terraform apply` |
| `bootstrap/` applied and outputs captured into GH repo secrets/vars | `envs/tst/` `terraform plan` via Actions |

## Reversal

If this decision turns out wrong, reversal is non-trivial but bounded:

- The Terraform state in the storage account is portable; it can be pulled
  and re-pushed against any backend.
- The naming standard already abstracts physical repo location, so renaming
  resources is not required.
- All git history would be preserved by `git filter-repo` back into the
  source repo if `wellnex-infra` were ever deprecated.

There is no point of no return inside the first three months of life.
